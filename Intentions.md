# AI News Dashboard - Project Intentions & Vision

> 이 문서는 프로젝트의 의도, 목적, 그리고 개발 방향을 기록합니다.
> 새로운 기능 개발이나 Claude와의 대화 시 참고 자료로 활용됩니다.

---

## 1. 프로젝트 비전

### 핵심 목표
**AI/기술 뉴스를 수집하고, AI로 요약한 뒤, 소셜 미디어용 콘텐츠를 자동 생성하는 개인 대시보드**

### 해결하고자 하는 문제
1. **정보 과부하** - AI 업계 뉴스가 너무 많아 따라가기 어려움
2. **콘텐츠 생성 피로** - 각 플랫폼별로 다른 스타일의 글을 작성하는 것이 시간 소모적
3. **일관성 유지 어려움** - 본인만의 글쓰기 스타일을 매번 유지하기 어려움

### 타겟 사용자
- AI/기술 분야 콘텐츠 크리에이터
- 소셜 미디어 매니저
- 기술 블로거 및 뉴스레터 운영자
- AI 업계 동향을 팔로우하는 개인

## 2. 핵심 워크플로우

```
[뉴스 소스 등록] → [RSS/URL 수집] → [AI 요약 생성] → [뉴스 리뷰] → [플랫폼 선택] → [콘텐츠 생성] → [피드백 반영] → [복사 & 게시]
```

### 상세 워크플로우

#### Phase 1: 설정 (1회성)
1. 뉴스 소스 추가 (RSS 또는 웹사이트 URL)
2. 본인 스타일의 글쓰기 템플릿 생성 (선택)

#### Phase 2: 수집 (주기적)
1. "Fetch All" 버튼으로 모든 활성 소스에서 뉴스 수집
2. 백그라운드에서 AI가 각 뉴스를 3줄 요약
3. 카테고리 자동 분류 (product, update, research, announcement, other)

#### Phase 3: 콘텐츠 생성 (필요시)
1. 뉴스 카드 클릭 → 상세 모달 열림
2. "Generate Content" 탭에서 플랫폼 선택 (Twitter, Threads, Instagram, LinkedIn)
3. 스타일 템플릿 적용 (선택)
4. AI가 플랫폼에 맞는 콘텐츠 생성
5. 피드백 입력 후 재생성 가능
6. 만족하면 복사하여 소셜 미디어에 게시

## 3. 구현된 기능 및 의도

### 3.1 뉴스 수집 시스템

| 기능 | 의도 | 구현 방식 |
|------|------|----------|
| RSS 피드 파싱 | 주요 AI 기업 블로그 자동 수집 | rss-parser 라이브러리 |
| URL 스크래핑 | RSS 없는 사이트도 수집 가능 | Cheerio로 본문 추출 |
| 중복 제거 | 같은 뉴스 중복 저장 방지 | URL 기반 dedupe |
| 소스 관리 | 관심 소스만 선택적 수집 | isActive 토글 |

**기본 소스 (의도적으로 AI 빅테크 중심):**
- OpenAI Blog
- Anthropic News
- Google AI Blog
- Meta AI Blog
- Hugging Face Blog

### 3.2 AI 요약 시스템

| 기능 | 의도 | 구현 방식 |
|------|------|----------|
| 3줄 핵심 요약 | 빠른 스캔을 위한 핵심만 추출 | DeepSeek AI |
| 카테고리 분류 | 관심 카테고리 필터링 가능 | AI 자동 분류 |
| 자동 요약 | 수집과 동시에 요약 생성 | 백그라운드 처리 |

**AI 프로바이더:** DeepSeek (deepseek-reasoner 모델)
- 비용 효율적
- 추론 특화 모델
- OpenAI 호환 API

### 3.3 콘텐츠 생성 시스템

| 플랫폼 | 글자 제한 | 스타일 특징 | 의도 |
|--------|----------|------------|------|
| Twitter/X | 280자 | 간결, 이모지 최소, 해시태그 1-2개 | 빠른 정보 전달 |
| Threads | 500자 | 캐주얼, 대화체 | 친근한 커뮤니케이션 |
| Instagram | 2200자 | 이모지 풍부, 해시태그 5-10개 | 발견성 최적화 |
| LinkedIn | 3000자 | 전문적, 인사이트 중심 | B2B 네트워킹 |

### 3.4 스타일 템플릿 시스템

**의도:** 매번 같은 톤을 유지하기 어려우므로, 기존 글 예시를 AI가 분석하여 스타일을 추출하고 재사용

| 기능 | 의도 |
|------|------|
| 예시 분석 | 3-5개 기존 글에서 톤과 특징 추출 |
| 템플릿 저장 | 플랫폼별 여러 스타일 관리 |
| 기본 템플릿 | 플랫폼별 기본 스타일 지정 |

### 3.5 피드백 시스템

**의도:** 한 번에 완벽한 콘텐츠를 기대하기 어려우므로, 피드백 기반 개선 루프 제공

```
[첫 생성] → "더 기술적으로" → [재생성] → "유머 추가" → [재생성] → [만족]
```

## 4. 기술적 결정 및 이유

### 4.1 상태 관리: Zustand + TanStack Query

**왜 이 조합인가?**
```
Zustand: 클라이언트 상태 (뉴스, 소스, 템플릿)
         → localStorage 자동 동기화
         → 동기적 hydration으로 깜빡임 없음

TanStack Query: 서버 상태 (API 호출)
         → 자동 캐싱, 재시도, 로딩 상태
         → mutation으로 낙관적 업데이트
```

**이전 방식의 문제:**
- localStorage 직접 접근 → 코드 중복, 일관성 없음
- useState + fetch → 로딩/에러 상태 직접 관리 필요

### 4.2 클라이언트 사이드 저장소

**왜 데이터베이스 없이?**
1. **프라이버시** - 뉴스 데이터가 서버로 안 감
2. **단순성** - 백엔드 인프라 불필요
3. **오프라인** - 한번 로드되면 오프라인 작동
4. **개인 사용** - 멀티유저 기능 불필요

### 4.3 Next.js 16 App Router

**선택 이유:**
- 서버 컴포넌트로 초기 로딩 최적화
- API Routes로 백엔드 로직 통합
- Turbopack으로 빠른 개발 경험

### 4.4 Tailwind CSS v4

**선택 이유:**
- 빠른 UI 프로토타이핑
- 다크 모드 기본 지원
- 번들 사이즈 최적화

### 4.5 DeepSeek AI

**선택 이유:**
- 비용 효율적 (GPT 대비 저렴)
- OpenAI 호환 API (기존 코드 재사용)
- 추론 특화 모델로 요약/분석에 적합

## 6. 향후 개발 아이디어

### 6.1 단기 (바로 구현 가능)

| 기능 | 설명 | 우선순위 |
|------|------|----------|
| 뉴스 북마크 | 나중에 볼 뉴스 저장 | 높음 |
| 다크 모드 토글 | UI에서 테마 전환 | 중간 |
| 내보내기 | 생성된 콘텐츠 히스토리 내보내기 | 중간 |
| 키보드 단축키 | j/k로 뉴스 탐색 | 낮음 |

### 6.2 중기 (설계 필요)

| 기능 | 설명 | 고려사항 |
|------|------|----------|
| 예약 게시 | 특정 시간에 게시 예약 | 서버 필요? 또는 알림만? |
| 콘텐츠 히스토리 | 생성된 모든 콘텐츠 저장 | 저장소 구조 설계 |
| 커스텀 프롬프트 | 플랫폼별 프롬프트 수정 | UI/UX 설계 |
| 다국어 생성 | 영어 뉴스 → 한국어 콘텐츠 | 번역 vs 재생성 |

### 6.3 장기 (대규모 변경)

| 기능 | 설명 | 고려사항 |
|------|------|----------|
| 소셜 미디어 직접 게시 | Twitter/LinkedIn API 연동 | OAuth 구현 필요 |
| 팀 협업 | 여러 사용자가 함께 사용 | 백엔드 DB 필요 |
| 멀티 AI 프로바이더 | Claude, GPT 등 선택 가능 | 프롬프트 통일 필요 |
| 분석 대시보드 | 게시물 성과 추적 | 각 플랫폼 API 필요 |
| 뉴스레터 생성 | 주간 AI 뉴스 요약 이메일 | 이메일 서비스 연동 |

## 7. 디자인 원칙

### UI/UX 원칙
1. **단순함 우선** - 필수 기능만 노출, 고급 기능은 숨김
2. **피드백 즉시** - 모든 액션에 토스트 알림
3. **반응형** - 모바일에서도 사용 가능
4. **빠른 로딩** - 1초 내 첫 화면 표시

### 코드 원칙
1. **타입 안전성** - TypeScript 엄격 모드
2. **관심사 분리** - UI, 상태, API 레이어 분리
3. **테스트 가능** - E2E 테스트로 주요 플로우 검증
4. **확장성** - 새 플랫폼 쉽게 추가 가능
